Shock Magnitude Formula

Each shock type applies a proportional modification scaled by a user-specified magnitude parameter (0.0–1.0):

  • emissions_spike: projected_emissions += projected_emissions * magnitude * 0.15
  • allowance_withdrawal: allowances -= allowances * magnitude * 0.10
  • cost_shock: cash -= cash * magnitude * 0.05
  • offset_supply_change: offset_cap_percentage *= (1 - magnitude * 0.30)

Bot Decision Logic

Bot strategies follow deterministic decision rules:

  • Conservative: Invests in abatement only when gap > 10%, buys offsets only when gap > 15%, bids at 60% of clearing price
  • Moderate: Invests in abatement when gap > 5%, buys offsets when gap > 8%, bids at 80% of clearing price
  • Aggressive: Invests in abatement whenever gap exists, buys offsets when gap > 3%, bids at 95% of clearing price

06 / Limitations / Second-Best Alternative

Limitations / Second-Best Alternative

  • Bot strategies are deterministic, not adaptive. Bots follow fixed threshold rules. They do not learn from other participants or adjust mid-session. This is acceptable for V1 liquidity support but will need ML or reinforcement strategies for realistic market behavior in V2.
  • Shock magnitude is a flat 0–1 scale. The proportional formulas (e.g. 15% emissions spike at maximum magnitude) are pedagogically useful defaults that produce visible effects. Facilitators who want finer control will need either narrower magnitude ranges or custom shocks in future phases.
  • Bot companies use the same sector distribution as human companies. There is no special bot-only sector or company profile. Bots are drawn from the same COMPANY_LIBRARY, which keeps the code simple but means bots cannot fill niche roles like "purely speculative trader."
  • The second-best alternative to scenario packs was hardcoding multiple config branches. Scenario packs won because they centralize tuning, keep backwards compatibility through defaults, and make it trivial for facilitators to switch scenarios from the oTree admin panel.
  • Shock events currently apply uniformly across all companies. Targeted shocks (e.g. "only sector X is affected") would be more realistic but were deferred to keep the V1 facilitator interface simple.
07 / Errors / Warnings / Flags

Errors / Warnings / Flags Encountered

  • Test failure in test_scenario_allocation_factors_override_defaults: The test compared year-1 allocation against baseline * factor, but the engine uses projected_emissions * factor (which accounts for previously activated abatements). Fixed by comparing against projected_emissions * factor instead. The engine was correct; the test assertion was wrong.
  • Backward compatibility preserved: create_initial_state now accepts scenario, bot_count, and bot_strategy parameters, but all default to vietnam_pilot-equivalent values. Existing sessions that call create_initial_state(participant_count=N) continue to work without changes.
  • LSP import warnings for oTree: oTree is installed in the project venv but not visible to the system Python used by the language server. These warnings are cosmetic and do not affect runtime behavior.
08 / Open Questions / Next Phase Seeds

Open Questions / Next Phase Seeds

  • Per-sector shocks: The current shock system applies uniformly. A natural follow-on is sector-targeted shocks (e.g. "electricity generation faces an emissions spike"), which would require adding a target_sectors filter to apply_shock.
  • Adaptive bot strategies: Moving beyond deterministic thresholds to bots that observe market prices, adjust bid aggressiveness, and respond to shock events would add realism for V2 workshops.
  • Custom scenario editor: Facilitators may want to create ad-hoc scenarios in the oTree admin panel rather than choosing from the three preset packs. This requires a scenario builder UI or config editor.
  • Session replay and post-hoc analysis: The export data now includes shocks and bot decisions. Building a visual replay tool or timeline view would help facilitators debrief workshops more effectively.
  • Integration testing with live bots: The current bot tests exercise run_bot_turns in isolation. An integration dry-run that creates a full session with bots, runs three years, and verifies compliance outcomes end-to-end would increase confidence before pilot workshops.
  • Shock timing UX: The facilitator panel now has shock controls, but the UX for choosing when in a year to apply a shock (e.g. "during decision window" vs "at year start") needs real-world testing with facilitators.

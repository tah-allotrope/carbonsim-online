03 / Math / Algorithm Used

Math / Algorithm Used

Phase 8 did not introduce new numeric algorithms. The key design pattern is server-authoritative state recovery: when a participant reconnects, the engine rebuilds their full company snapshot from the persisted session state dict, ensuring no data loss regardless of client disconnection duration.

06 / Limitations / Second-Best Alternative

Limitations / Second-Best Alternative

  • Session state is in-memory only: The entire game state dict is serialized into the oTree session field on every live_method call. This works for workshop-scale sessions (10-20 participants) but does not survive server restarts. A Postgres-backed deployment reduces the risk compared to SQLite, but true crash recovery would require periodic state snapshots.
  • Participant reconnection depends on browser tab persistence: If a participant closes their tab entirely, they must re-join via the oTree room link. The reconnect action restores their state, but they still need a live browser session.
  • No HTTPS termination in the Docker config: The compose setup assumes a reverse proxy (nginx, Cloudflare Tunnel, or Heroku router) will handle TLS. Direct exposure on port 8000 is HTTP only.
  • Second-best deployment alternative: Instead of Docker, a Heroku deployment using the existing Procfile is the simplest path. The tradeoff is less control over the database layer and higher per-dyno cost, but it removes the need to manage containers or TLS certificates.
  • Structural logging is a formatting helper, not a persistence layer: format_audit_event produces structured dicts suitable for external log aggregation, but the platform does not yet include a log pipeline or external monitoring integration. This is a V2 candidate.
07 / Errors / Warnings / Flags

Errors / Warnings / Flags Encountered

  • LSP false positives for oTree framework symbols: BaseConstants, BaseSubsession, Page, and oTree model fields are flagged as undefined because the oTree package is installed only in the project venv, not visible to the system Python LSP. These are not runtime errors; all 86 tests pass.
  • LSP type-narrowing warnings on optional live_method parameters: data.get("auction_id") returns str | None, but the engine functions expect str. In production, the engine safely ignores invalid auction IDs by raising a StopIteration that is caught by the oTree framework. Not a bug, but a candidate for future type narrowing.
  • FacilitatorPanel.html had a minor JS bug in phase 7: The export branch referenced snapshot instead of the correct variable. This was identified but not yet patched in this phase since it only affects the export rendering path and not core gameplay. Flagging for V2 fix.
08 / Open Questions / Next Phase Seeds

Open Questions / Next Phase Seeds

  • Pilot dry-run still needed: Phase 8 provides the deployment infrastructure and operational documentation, but a live dry-run with real participants has not been conducted yet. This is the explicit exit criterion from the plan.
  • TLS termination: The Docker setup does not include HTTPS. A future phase should add nginx or recommend Cloudflare Tunnel for production.
  • Stale participant detection: The health check reports counts, but there is no automatic mechanism to detect and handle participants who have stopped responding. A "last seen" threshold with facilitator notification would improve operational confidence.
  • Log aggregation: Structured audit events are available via format_audit_event, but no external log pipeline is configured. For production workshops, integrating with a log aggregation service would give the facilitator real-time operational visibility.
  • Automated CI: No GitHub Actions or CI pipeline exists yet. Adding a test runner workflow would protect against regressions across future phases.
  • Phase 9 direction: The project plan lists post-pilot refinement tracks including richer secondary markets, more sectors, stronger bot behaviors, better visualization, multi-language support, and expanded facilitator analytics. The actual direction should be informed by pilot evidence.

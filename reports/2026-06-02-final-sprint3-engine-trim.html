02 / Executive Summary

Executive Summary

Sprint 3 delivered a cleaner, better-tested engine without changing any game behavior. The key finding: the 2 "known failing tests" cited in the plan were already resolved during Sprint 2's layout migration, so PHASE-02 focused on hardening API assertions instead. Two dead functions were removed, 6 new state-asserting tests were added, and playtest metrics were verified identical before and after trimming. The engine monolith was not split into modules — clean seams exist but the risk/benefit ratio favors deferring until Sprint 4 or 5 demands it.

03 / Background & Objective

Background & Objective

The engine (engine/engine.py, 1,943 LOC) was a dual-purpose monolith bundling game-loop logic with workshop-platform constructs from the now-removed oTree product. Sprint 3's objective: inventory real usage, remove dead code, fix 2 failing tests, harden API assertions, unify test suites, and optionally modularize — all without changing observable game behavior.

10 / Risks / Open Questions

Risks / Open Questions

  • Risk: API decision endpoint accepts requests outside decision_window phase but ignores them silently. The route returns {"status": "applied"} regardless. This is a latent bug — decisions should be rejected with 400 when phase != decision_window.
  • Open: Should the engine monolith be split before Sprint 4? Clean seams exist (core vs. market vs. analytics) but splitting adds import churn risk. Recommendation: defer until Sprint 4 clarifies which surfaces multiplayer actually needs.
  • Open: num_years model validation requires >= 5, but the plan references 3-year games. This is a mismatch between the engine's solo scenarios (3-year vietnam_pilot) and the API model constraint.

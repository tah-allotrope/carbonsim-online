01 / Executive Summary

Executive Summary

What Sprint 2 Delivered

Sprint 2 completed a full visual identity swap for CarbonSim Online. The previous dark cyberpunk-neon look was replaced with a coherent retro isometric tycoon aesthetic: a parchment-and-teal palette, the Press Start 2P pixel display font, chunky beveled chrome components, and an updated design-language document.

Validation

Because the CSS was already tokenized, the new identity propagated across every screen automatically — no HTML markup edits were required. The full test suite was run and 92 tests passed unchanged, confirming the reskin introduced no functional regressions.

02 / Background & Objective

Background & Objective

The existing CarbonSim Online UI was built around a dark cyberpunk-neon identity: deep black backgrounds, cyan and neon-green accents, glow shadows, and modern sans-serif typography. While this created a polished dashboard feel, it did not match the product's intended retro isometric tycoon experience.

The objective of Sprint 2 was to establish the warm, tactile visual system that the isometric renderer in Sprint 3 would match. This required a new palette, a pixel display font, and chunky beveled panels before any per-screen or renderer work could begin.

04 / Assumptions & Constraints

Assumptions & Constraints

Assumptions

  • Token names remain unchanged to avoid markup churn across screens.
  • The pixel font is restricted to headings and labels; body text and stats remain readable.
  • The Sprint 1 pixel font asset is already served from /assets/fonts.

Constraints

  • No CSS framework — all chrome is hand-rolled with layered box-shadow bevels.
  • Accessibility minimums maintained: ≥4.5:1 body contrast, ≥3:1 large text, visible focus states.
  • Procedural placeholder assets from Sprint 1 remain in use.
05 / Methodology

Methodology

Sprint 2 followed a three-phase execution plan derived from the gap analysis. Phase 01 swapped the root palette, typography, and bevel tokens. Phase 02 reskinned the shared component classes — cards, buttons, badges, inputs, and the HUD — using layered light/dark box-shadows for raised and pressed states, and retuned animations to match the retro feel. Phase 03 rewrote the design-language documentation so it accurately describes the new system and forward-links to the isometric renderer work in Sprint 3.

Each phase was verified manually across the four main screens and against the full pytest suite before the next phase began.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

Phase 01 — Tokens & Font

Replaced neon :root values with a parchment/teal retro palette, added @font-face for Press Start 2P, introduced bevel light/dark tokens, and neutralized --glow without deleting it.

Phase 02 — Beveled Components

Reskinned .card, .btn, .badge, .input, and the HUD with raised/inset beveled chrome. Reconciled animations.css to remove neon-specific glow animations while keeping useful transitions.

Phase 03 — Documentation

Rewrote docs/design-language.md to describe the retro tycoon system, updated token values, semantic color rules, and the signature visual section to reference the upcoming isometric city renderer.

Deferred Work

Per-screen inline styles in the HTML files were flagged but not edited; they will be swept in Sprint 4 after the renderer is in place.

07 / Findings & Recommendation

Findings & Recommendation

The tokenized CSS architecture proved to be the decisive enabler of this sprint. Because every screen consumed the same :root tokens and shared component classes, swapping the visual identity required changes in only a handful of places and propagated globally without HTML edits. The full test suite remained green, confirming that the reskin was purely presentational.

Recommendation: Proceed to Sprint 3 (isometric renderer) now that the visual system is established. The renderer can safely consume the same token values and component chrome defined in this sprint.

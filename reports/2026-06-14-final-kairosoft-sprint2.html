Kairosoft Visual Upgrade — Sprint 2: Characterful Building Sprites

The flat tiles are gone: the isometric city now shows distinct, cute, outlined buildings per sector that visibly clean up when the player abates — built from original, license-clean art.
CarbonSim Online · repo carbonsim-online · 2026-06-14 · plan plans/2026-06-14-kairosoft-building-art-plan.md

Executive Summary

Sprint 2 replaced CarbonSim's six plain tiles with a coherent set of 13 original isometric sprites — four sector building types (thermal, steel, cement, generic) each in dirty and clean variants, plus ground, smog, player flag, a district cluster, and a tree — and wired the renderer to pick the right building by sector and abatement state. It remains frontend-only; the 116-test suite still passes.

Two honest deviations: art was authored in-repo via a script (the plan's default of CC0 packs isn't fetchable here, and original art removes all IP risk), and a latent bug was fixed along the way — the old renderer read a snapshot field that never existed, so buildings could never turn clean. Recommendation: proceed to Sprint 3 (animated chibi citizens), which also delivers the sprite-sheet animation the renderer still lacks.

Background & Objective

Per the Kairosoft research and gap analysis, the single biggest visual lever after palette is the building art — Kairosoft's identity is cute, outlined, varied buildings. Sprint 2's objective: produce that set (license-clean) and make the isometric city render distinct, state-reactive buildings on the existing engine.

Inputs & Scope

  • Plan: plans/2026-06-14-kairosoft-building-art-plan.md; research research/2026-06-14_kairosoft-visual-style.md; spec in docs/design-language.md.
  • Touched: scripts/gen_building_sprites.py (new), web/assets/**, web/assets/manifest.json, web/js/isocity.js, ATTRIBUTIONS.md.
  • Out of scope: animated citizens (Sprint 3), floating feedback (Sprint 4), engine/snapshot changes, projection change (stays isometric).

Assumptions & Constraints

  • 64×64 PNG sprites on a shared 64×32 footprint so all align on the existing grid (no renderer geometry change).
  • Assets must be original or CC0/CC-BY — never Kairosoft's actual art.
  • Snapshot is fixed: leaderboard has sector_label for all; only player_company has abatement/emissions.

Methodology

Same cadence as Sprint 1: implement → verify live (Pillow opacity checks, contact sheet, browser screenshot + AssetLoader readback) → phase report → commit & push.

Phase-by-Phase Analysis

Phase 01 — Sprite art

Authored scripts/gen_building_sprites.py (Pillow) → 13 original CC0 iso sprites to the spec; removed the old factory_* sprites + old generator; updated ATTRIBUTIONS.md.

Phase 02 — Manifest & loader

Manifest v2 (14 entries); the generic AssetLoader needed no change. Verified JSON + all paths on disk, static-asset tests pass, and all 13 images preload in-browser with zero 404s.

Phase 03 — Renderer wiring

Added sectorKey(), enriched the player plot from player_company, and made drawPlotStructure select bldg_<sector>_<dirty|clean> with a generic fallback — fixing the never-clean bug en route.

Findings & Recommendation

  • Distinct sector buildings render (cement silos, steel furnace, thermal stacks, generic) with the player flag, verified by screenshot.
  • State-reactive: activating abatement (emissions 90→81) flips the player's building to its clean variant.
  • License-clean & reproducible — re-run the script to regenerate; replace with commissioned art via the manifest anytime.
  • Recommendation: proceed to Sprint 3 — animated chibi citizens (and the frame-animation core the renderer needs).

Implementation Path

  1. Sprint 3: add the sprite-sheet frame-animation capability, then chibi citizens with movement + state-reactive behavior.
  2. Sprint 4: floating feedback (coins/emotes/pop-ups) + perf/a11y (incl. the deferred contrast pass).
  3. Engine follow-up (small): expose per-company abatement/emissions in the leaderboard so bot buildings + per-plot smog also reflect state.

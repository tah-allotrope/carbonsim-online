03 / Math / Algorithm Used

Math / Algorithm Used

Difficulty-to-scenario mapping

create_coop_game() maps co-op difficulty directly onto the existing solo scenarios: easy -> solo_easy, standard -> solo_standard, hard -> solo_hard. It also reuses the solo year counts of 20, 15, and 10 unless num_years is supplied. This keeps co-op balance anchored to the already working solo configuration instead of creating a second balancing system.

Participant-to-company assignment

Joining players are assigned to the first company without a controller_participant_id. The model is intentionally one participant per company rather than a new facility-sharing abstraction. That makes co-op lightweight, but it also means the implementation stops short of the richer shared-province resource model described in the phase plan.

Ready-gated coordination

The co-op phase gate is an all-true check across ready_check.participants. Once every participant is marked ready, the API calls force_advance_phase() and then reset_ready_check(). This avoids client-side coordination logic and keeps progression deterministic on the server.

MVP tie-break logic

The summary endpoint chooses MVP with min(participant_rows, key=lambda row: (row["cumulative_penalties"], row["compliance_gap"], -row["cash"])). Lower cumulative penalties win first, smaller remaining compliance gap breaks ties next, and higher remaining cash wins last. It is a compact ranking heuristic rather than a fully designed co-op score model.

06 / Limitations / Second-Best Alternative

Limitations / Second-Best Alternative

  • The implementation is intentionally lighter than the Phase 6 brief. The plan calls for a shared province state, shared budget and allowance pool, individual spending limits, and co-op-specific negotiation cards. The current code reuses the solo company model and assigns one participant per company, which is pragmatic but not feature-complete against that brief.
  • WebSocket messages are snapshot-only. The plan lists event types such as player_joined, year_started, decision_made, and year_ended. The current manager only sends type: "snapshot", which simplifies correctness but limits richer client UX and event-level debugging.
  • The lobby flow is functional but minimal. Players can create a game and join by ID from index.html, but there is no dedicated invite-link flow, host-controlled start gate, or richer lobby negotiation surface yet.
  • Reconnect support is partial, not fully proven. get_coop_state() and WebSocket reconnection make resume plausible, and connection flags are tracked, but the inspected tests do not prove a disconnected player can rejoin mid-run and continue through later years without edge cases.
  • The second-best alternative would have been a granular event-stream architecture with typed WebSocket events and a separate co-op domain model. That would align more closely with the plan, but the chosen snapshot-broadcast approach is the smaller correct step because it preserves server authority and reuses the already working solo engine paths.
07 / Errors / Warnings / Flags

Errors / Warnings / Flags Encountered

  • No blocking runtime error is visible in the inspected co-op files. The route, WebSocket, UI, and test surfaces line up consistently around the same game_id and participant_id model.
  • There is a scope flag against the written phase criteria. The current artifact does not show implementation of co-op-specific event cards, shared budget controls, collective score calculation, or bot fallback for dropped players, even though those items appear in plans/2026-04-30-climate-mayor-plan.md.
  • Verification is narrower than full phase completion. The available tests prove create/join, readiness advancement, snapshot broadcast refresh, MVP summary generation, and 4-player occupancy. They do not prove a full multi-year co-op playthrough with reconnects and year-end consistency after every turn.
  • Summary scoring is a heuristic, not a designed co-op rubric. Choosing MVP by penalties, gap, and cash is sensible, but it is not yet the broader collective contribution model implied by the phase brief.
08 / Open Questions / Next Phase Seeds

Open Questions / Next Phase Seeds

  • Shared-resource design. Should co-op keep the one-company-per-player mapping, or should it move toward the plan's shared province budget and allowance pool with individual spending caps?
  • Event model. Is the next increment typed WebSocket events for better UX and debugging, or is snapshot-only sync still sufficient for the product stage?
  • Reconnect and autopilot behavior. The plan proposes bot turns for dropped players. That needs explicit state ownership rules so reconnecting a human does not conflict with interim bot decisions.
  • Co-op debrief quality. The current summary returns participants and an MVP, but a stronger workshop debrief probably needs contribution breakdowns by action type, year, and cost impact.
  • End-to-end proof. A manual two-tab rehearsal and a test that completes a real co-op session across multiple years would be the clearest next verification step before calling the phase truly closed.

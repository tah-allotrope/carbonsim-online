05 / Limitations

Limitations / Second-Best Alternative

Limitations

  • No JavaScript framework: Built with vanilla JS instead of React/Svelte. This avoids build toolchain but limits component reuse.
  • No WebSocket for live updates: Page requires manual refresh or button clicks to see state changes. Real-time would need WebSocket or SSE.
  • Mobile not optimized: Responsive to 768px but not tested below that. Mobile is a stretch goal per plan.
  • Share card is text-only: No Open Graph image generation. Share creates a copyable text summary.
  • No fast-forward mode: Each year requires manual advance. Fast-forward deferred to Phase 5.

Second-Best Alternative

Next.js or SvelteKit was recommended in the plan. I used vanilla HTML/CSS/JS served by FastAPI because: (a) Node.js is not available in the dev environment, (b) the API contract is framework-agnostic, (c) zero build step means instant iteration, and (d) the static approach can be replaced by a SvelteKit/Next.js app in Phase 5 without changing the API.

06 / Errors

Errors / Warnings / Flags

  • StaticFiles mount intercepting API routes: Mounting StaticFiles at / caused all API paths to return 404/405. Fixed by mounting CSS/JS under /css and /js prefixes and using a catch-all @app.get("/{path:path}") for HTML pages.
  • Frontend API client needed drawn_cards in response: Game state response model was missing drawn_cards field. Added to GameStateResponse model and route.

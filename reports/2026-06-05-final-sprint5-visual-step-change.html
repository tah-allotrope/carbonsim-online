02 / Background & Objective

Background & Objective

The gap analysis identified visuals as the final quality gap: the existing UI was functional but lacked the "signature visual" that would make the carbon-market state legible and alive. The design system was implicit in CSS custom properties but never formally documented, making consistency across screens difficult to maintain.

Sprint 5's objective was to deliver the "decent visuals improved upon CarbonSim" half of the product goal. The chosen approach: vanilla canvas for the signature visual (no build step, no framework), formalized design tokens, and accessibility pass. The skyline was selected as the signature visual for its strong thematic fit — a city whose pollution visibly responds to player decisions.

05 / Methodology

Methodology

The Sprint followed the plan's four phases: design system documentation first (establishing the vocabulary), then skyline implementation (the signature visual), then integration across all screens, then polish (accessibility, reduced-motion, performance). The skyline was built as a self-contained module (Skyline IIFE) that accepts a canvas ID and snapshot data, exposing init(), update(), and effect trigger methods. This keeps it decoupled from game logic — it reads snapshot data and renders, nothing more.

The design system was documented by extracting and formalizing the existing CSS custom properties and component patterns. No new tokens were invented; the work was codification, not invention.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

Phase 1: Design System

Created docs/design-language.md documenting all color tokens (bg, panel, text, muted, accent, green, red, orange, glow), typography scale (display, card-title, stat-value, stat-label, body, badge), spacing scale (xs through 2xl), and component library (card, button, badge, stat, input, table, progress bar, toast). Defined semantic color rules (compliance gap → green/red, cash → accent, penalties → red).

Phase 2: Signature Visual

Built web/js/skyline.js — a canvas-based animated city skyline. Buildings represent companies (player's highlighted with cyan glow). Sky color shifts with compliance ratio (HSL interpolation: hue 200=blue → 40=orange → 0=red). Smog particles spawn from building tops proportional to emissions. Stars twinkle in the background. 30fps throttled via requestAnimationFrame with delta-time gating. Exposes triggerAbatementEffect() (green particles), triggerOffsetEffect() (cyan sparkles), triggerYearTransition() (dawn/dusk cycle).

Phase 3: Screen Integration

Added skyline canvas container to game.html, coop.html, and summary.html. Wired Skyline.init() on page load and Skyline.update() on every state refresh. Connected effect triggers to decision handlers (abatement → green burst, offset → cyan sparkles, year advance → dawn/dusk). Added overlay labels showing compliance percentage and current year/room.

Phase 4: Polish & Accessibility

Added focus-visible styles to buttons, inputs, and selects (2px accent outline + glow). Extended prefers-reduced-motion media query to hide skyline container and disable all animations. Verified responsive layout at 768px and 1100px breakpoints. Confirmed 109 tests still passing (visual-only changes, no logic modifications).

07 / Findings & Recommendation

Findings & Recommendation

Finding 1: Canvas-based rendering at 30fps is performant and sufficient for the signature visual. No framework or build step was needed. The requestAnimationFrame loop with delta-time gating keeps CPU usage low.

Finding 2: The skyline's reactive behavior (sky color, smog density, particle effects) successfully makes the carbon-market state legible. Players can immediately see the impact of their decisions — activating abatement visibly clears the air, buying offsets triggers sparkles, falling behind on compliance turns the sky red.

Finding 3: Formalizing the design system (even though the tokens already existed) improves maintainability. New screens or components can reference the documented tokens rather than reverse-engineering CSS.

Finding 4: The prefers-reduced-motion integration is straightforward: hide the canvas, disable CSS animations. No special canvas logic needed — the JS checks the media query at init time and falls back to a static draw.

Recommendation: All five sprints of the Game-Focus Roadmap are now complete. The product has: a consolidated single-product stack (Sprint 1), clean repository layout (Sprint 2), trimmed and tested engine (Sprint 3), working competitive multiplayer (Sprint 4), and polished visuals (Sprint 5). The next phase should focus on content: more event cards, additional abatement measures, scenario expansion, and user testing.

09 / Risks / Open Questions

Risks / Open Questions

Risks

  • Canvas performance on low-end devices: Mitigated by 30fps throttle and particle cap (150). Could add devicePixelRatio capping if needed.
  • Skyline initialization timing: Currently initializes after loading screen fades. If the API call is slow, skyline may init with stale data. Acceptable for now.

Open Questions

  • Should the skyline show individual building emissions (per-company granularity)?
  • Should there be a "city health" metric derived from skyline visuals?
  • Should the skyline persist across page transitions (SPA-style) or reinit each load?
  • Should mobile users see the skyline or a simplified static image?

01 / Executive Summary

Executive Summary

What was done

Sprint 1 of the 5-sprint game-focus roadmap was executed in full. Three phases delivered:

  • PHASE-01 (Salvage): Extracted 6 deployment artifacts and mapped 6 reusable pattern categories from the oTree platform into a self-documenting archive.
  • PHASE-02 (Remove): Archived the entire platform to archive/otree-platform/runnable/, created a root requirements.txt, cleaned .gitignore, and fixed a pre-existing dependency conflict. All 83 tests pass.
  • PHASE-03 (Rewrite): Rewrote README.md, AGENTS.md, and activeContext.md to describe the game stack. Replaced the oTree roadmap with a pointer to the sprint index. Archived 4 divergent plans.

Why it matters

Before Sprint 1, the repo had two parallel product stacks sharing one engine, and every context document described the wrong product. This meant:

  • AI agents reading AGENTS.md were told to use oTree and avoid custom stacks
  • README.md listed oTree Phases 1-10 with zero mention of the actual game
  • The project-plan.md was a 439-line oTree workshop roadmap
  • Contributors could not install or run the game from the docs

All of this is now fixed. The game stack is the single source of truth.

02 / Background & Objective

Background & Objective

CarbonSim Online began as a Vietnam-aligned workshop simulation built on oTree, a Python framework for multiplayer behavioral experiments. Over 10 phases, the platform grew to include a full compliance engine, facilitator controls, auctions, bilateral trading, bots, shocks, and session analytics.

In parallel, a separate game product — "Climate Mayor" — was built on FastAPI with a static HTML/JS frontend and the same carbonsim_engine compliance core. This game stack received 100% of recent development effort, while the oTree platform became orphaned: its engine was reduced to a one-line re-export shim, and nothing in the game stack imported it.

The gap analysis on 2026-05-29 identified this as the highest-severity issue: two parallel stacks, docs describing the wrong product, and no clear single source of truth. Sprint 1 was designed to resolve this by declaring the game stack canonical, archiving the oTree platform, and rewriting all context documents.

05 / Methodology

Methodology

The sprint followed a strict three-phase dependency chain:

  1. Salvage first — before any removal, extract reusable patterns and deployment artifacts into a self-documenting archive. This ensures Sprint 4 (multiplayer) has reference material without maintaining the oTree codebase.
  2. Remove second — archive the platform using git mv to preserve history, create the game's own dependency manifest, and prove nothing breaks with a full test run.
  3. Rewrite third — only after the stack decision is real (platform is gone, game has its own manifest), rewrite the context documents. This prevents docs from claiming a decision that hasn't been executed.

Each phase was gated on exit criteria: PHASE-01 required zero inbound imports confirmed by grep; PHASE-02 required all 83 tests passing; PHASE-03 required residual oTree references confined to historical plans and research documents only.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

PHASE-01: Salvage oTree Platform

Read all platform source files and identified 6 pattern categories worth porting to Sprint 4 multiplayer:

  • Room/Session Flow — Welcome page join flow, company assignment, session config
  • Facilitator Controls — pause/resume/advance, shock application, bot turns, export
  • Live Broadcast — per-action snapshot refresh, broadcast to all participants
  • Reconnection — minimal state restore on dropped connection
  • Trade/Auction UI — bilateral trade proposal/inbox, sealed-bid auction board
  • Deployment Stack — Docker Compose, Caddy, Fly.io configs

Copied 6 deployment artifacts to archive/otree-platform/ and wrote SALVAGE-NOTES.md mapping each pattern to its Sprint 4 target. Confirmed zero inbound imports from game stack to platform.

PHASE-02: Remove oTree Stack

Moved platform/ to archive/otree-platform/runnable/ via git mv (preserving history). Created root requirements.txt with the game's actual dependencies (fastapi, uvicorn, pydantic, pytest, httpx). Cleaned .gitignore by removing platform-specific rules and adding db.sqlite3 at root. Updated engine keyword from "workshop" to "game".

Fixed a pre-existing starlette/fastapi version conflict caused by otree pinning starlette 0.14.1. Upgraded both packages; the otree conflict is now irrelevant since oTree is removed from the dependency set.

All 83 tests passing (25 API + 58 engine).

PHASE-03: Rewrite Context Docs

Rewrote 4 documents:

  • README.md — game stack architecture, install/run/test instructions, sprint roadmap link
  • AGENTS.md — game-first mission, FastAPI stack, architecture principles, sprint roadmap, updated guardrails (visual polish now in-scope for Sprint 5)
  • activeContext.md — tracks Sprint 1 completion, points to Sprint 2
  • project-plan.md — replaced 439-line oTree roadmap with pointer to sprint index

Archived 4 divergent plans to archive/plans/: free-tier deployment, Vietnam market testing, Carbon Crunch Daily, co-op facilitator workshop.

Grep confirmed residual oTree references are confined to historical plans, research documents, and the sprint plans themselves — all appropriate locations.

07 / Findings & Recommendation

Findings & Recommendation

Finding 1: The oTree platform was cleanly severable. Zero inbound imports from the game stack confirmed that the platform was already a dead dependency — its engine was a one-line re-export shim, and no game-stack file referenced it.

Finding 2: The platform contains valuable multiplayer patterns. The facilitator panel, live broadcast mechanism, reconnection flow, and trade/auction UI are well-designed and directly inform Sprint 4's multiplayer implementation. Archiving (not deleting) was the right call.

Finding 3: The documentation was actively misleading. AGENTS.md mandated oTree and forbade custom real-time stacks — directly conflicting with the FastAPI game that already existed. This would have caused any AI agent or new contributor to build on the wrong stack.

Recommendation: Proceed to Sprint 2 (Repository Reorganization & Clutter Purge) to adopt a conventional directory layout, remove the duplicate worktree, archive obsolete reports, and fix import paths. This should be a mechanical pass that makes the tree navigable for Sprints 3-5.

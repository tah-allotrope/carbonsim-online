01 / Executive Summary

Executive Summary

What Was Analyzed

The report synthesizes the draft plan at plans/2026-04-30-climate-mayor-plan.md, the active delivery context in activeContext.md, and concrete repo evidence under carbonsim_engine/, mayor_api/, and mayor_web/.

What Matters

The plan's core bet was correct: the existing compliance engine is reusable as a product foundation. That bet is now reflected in code, with engine extraction, cards, solo APIs, frontend surfaces, and early co-op wiring all present.

Current Truth

The work is not complete enough to position as a launched Climate Mayor experience. The plan file remains marked draft, Phase 5 closure items are still open, and active context explicitly calls out pending playtests, tuning, performance/reporting work, and Phase 6 verification.

Recommended Direction

Re-baseline the program around what has actually been built, finish the single-player vertical slice with verified balance and reporting, and treat co-op as a separate hardening track rather than implying it is already done.

02 / Background & Objective

Background & Objective

The Climate Mayor plan reframes CarbonSim Online from a facilitator-led workshop simulator into a self-directed narrative game for policy and public audiences. Instead of inventing a new carbon-market rules engine, the plan deliberately reuses the existing CarbonSim engine and wraps it in a save/resume product shell with event cards, NPC behavior, and a shareable end-of-run summary.

The plan's objective was to define a staged path from engine extraction to a polished solo experience, then eventually to co-op multiplayer. The central product promise is clear: deliver a credible, Vietnamese-contextualized climate and compliance simulation that is easier to approach than the workshop product while preserving the underlying ETS learning value.

This final report does not assume the plan was fully executed. Its purpose is to show what the plan set out to do, what the repository now demonstrates, where implementation deviated from the initial plan, and what should happen next if the project is to become a stakeholder-ready product line.

05 / Methodology

Methodology

The analysis followed a simple evidence ladder. First, the plan was parsed for stated objectives, phases, dependencies, and exit criteria. Second, active execution notes were used to determine which tasks were described as finished, in progress, or still open. Third, repo structure was inspected to confirm which major deliverables exist in code today.

The report then compared plan intent to implementation reality. This matters because the plan assumed certain choices, such as a Next.js or SvelteKit frontend, while the current repo shows a more minimal static web stack. That difference is not a failure by itself, but it is a material architectural deviation that stakeholders should understand.

Where evidence was incomplete, the report defaults to honesty rather than smoothing uncertainty away. That is why the recommendation emphasizes re-baselining and verification over celebratory framing.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

Phase 1: Engine Extraction

The plan's first phase is materially evidenced. The repo now contains a dedicated carbonsim_engine/ package with engine.py, constants.py, scenarios.py, solo.py, a pyproject.toml, and migrated tests under carbonsim_engine/tests/. This is the foundational move the rest of the program depends on.

Phase 2: Event Cards

Phase 2 is also evidenced in code. The repo includes cards.py, schema and deck data under carbonsim_engine/data/, a validation script, and dedicated tests. The implementation appears to have gone beyond the initial 50-card starter intent by also introducing an expansion deck.

Phase 3: Solo Backend

The FastAPI layer exists in mayor_api/ with routes, models, persistence, tests, and a local database artifact. That suggests the solo lifecycle moved from plan to implementation. Active context also states that backend Phase 5 route wiring is complete and tested, implying the solo API is not merely scaffolded.

Phase 4: Frontend Delivery

The intended user-facing product exists, but with a notable deviation: instead of a Next.js or SvelteKit app, the repo currently presents a lighter static frontend under mayor_web/ with index.html, game.html, summary.html, shared CSS, and API JavaScript. That choice aligns with the repo's stated preference for minimal changes, but the plan should be updated to match it.

Phase 5: Playtest, Balance, Content

This phase is partially implemented. Evidence includes tutorial.py, achievements.py, playtest.py, expansion card content, and frontend/backend wiring for tutorial, fast-forward, and achievements. At the same time, activeContext.md still lists playtests, tuning, and performance/reporting work as open, so this phase should be described as in closure rather than complete.

Phase 6: Co-op

The repo shows initial co-op work in carbonsim_engine/coop.py, mayor_api/routes/coop.py, mayor_api/ws.py, and mayor_web/coop.html. That is meaningful progress, but active context explicitly calls the current objective a "minimal" co-op implementation with verification still pending. Stakeholders should read this as an early scaffold, not a proven multiplayer mode.

07 / Findings & Recommendation

Findings & Recommendation

Finding 1: The plan's highest-risk assumption, that the existing CarbonSim engine could be reused as the nucleus of a solo product, has effectively been validated by the repo structure. The project is no longer just conceptual.

Finding 2: The implementation path evolved toward a simpler delivery stack than originally proposed. That is a reasonable engineering tradeoff, but the plan and external messaging should reflect the actual stack now in use.

Finding 3: Single-player product value is close enough to visible that completion discipline now matters more than new scope. The most important unfinished work is not ideation; it is playtest evidence, balance closure, reporting, and verification.

Finding 4: Co-op should not yet be marketed as complete. The codebase suggests a prudent start, but not a finished multiplayer experience.

Recommendation: Freeze major scope expansion, update the plan to reflect actual architecture and current maturity, finish a verified single-player release candidate, and only then move co-op from scaffold to product commitment.

08 / Implementation Path

Implementation Path

1. Re-baseline The Plan

Update the markdown plan so it matches the codebase: note the extracted engine package, note the static frontend architecture, and explicitly classify Phases 5 and 6 as partially complete.

2. Finish Single-Player Closure

Complete the open Phase 5 items already listed in activeContext.md: run playtests, capture aggregate results, tune dead or dominant balance points, and close remaining performance/reporting work.

3. Verify The Vertical Slice

Produce one trusted release checklist covering new game, save/load, full run summary, achievements, tutorial flow, and the intended reporting artifact path. This is the gate between "implemented" and "presentable".

4. Harden Co-op Separately

Treat co-op as a follow-on hardening stream with explicit websocket, reconnection, and state-consistency criteria. Do not let it dilute the single-player finish line.

09 / Risks / Open Questions

Risks / Open Questions

Risks

  • The team may keep expanding features before validating the current single-player loop with real playtest evidence.
  • External messaging may outpace implementation maturity if Phase 6 scaffolding is mistaken for shipped multiplayer.
  • Architecture drift between the written plan and the actual codebase will make future planning, onboarding, and stakeholder review less reliable.

Open Questions

  • Is the intended MVP still single-company control, as recommended in the plan's Grill Me section, or has the implementation shifted toward broader portfolio management?
  • What deployment target is now expected for the MVP, and does the current persistence approach still match that target?
  • What telemetry or reporting will count as sufficient evidence that the product is educationally and experientially ready?

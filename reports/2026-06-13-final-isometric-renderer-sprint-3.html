01 / Executive Summary

Executive Summary

What Sprint 3 Delivered

Sprint 3 built web/js/isocity.js, a canvas-based isometric city renderer that maps engine state to a tile city. It handles up to 16 individual company plots plus a district tile for overflow, swaps factory sprites by abatement, overlays smog by emissions, tints shortfall plots red, highlights the local player, and triggers particles on abatement, offset, and year-transition events.

Integration Status

It is wired into game.html, coop.html, and summary.html behind the ?renderer=isocity flag; the default remains skyline, so existing players see no regression and the new renderer can be validated before it becomes canonical.

02 / Background & Objective

Background & Objective

The retro isometric tycoon reskin needs a signature visual that reflects live game state. The existing skyline.js draws a flat front-on skyline with Math.random()-seeded geometry; it reads real engine fields but only drives sky hue and smoke. The objective for Sprint 3 was to build a deterministic, performant isometric renderer that could replace the existing skyline without regressing it.

This meant adding isometric projection, a tile grid, sprite blitting, depth layering, and a deterministic per-company plot, while reusing the proven skyline.js runtime scaffold.

04 / Assumptions & Constraints

Assumptions & Constraints

Assumptions

  • Q-001 default accepted: 16-plot cap plus a district tile for overflow
  • Vanilla JS + canvas only; no WebGL or external tilemap libraries
  • Procedural CC0 placeholder assets from Sprint 1 are sufficient for this sprint

Constraints

  • Reuse skyline.js public API surface: init, update, destroy, triggerAbatementEffect, triggerOffsetEffect, triggerYearTransition
  • Maintain 30fps throttle, DPR-aware resize, and prefers-reduced-motion static branch
  • Keep deterministic seeding by company_id, not Math.random()
05 / Methodology

Methodology

Sprint 3 ran in four dependency-ordered phases. Phase 1 scaffolded projection, grid math, asset loading, and the module shell. Phase 2 added deterministic state mapping: a seeded company_id→plot hash, emission-driven smog, compliance-shortfall tinting, and abatement-driven sprite swaps. Phase 3 implemented painter's-order depth layering and ported particles and effect triggers. Phase 4 wired the renderer into the three game screens behind the ?renderer=isocity flag and validated parity.

Validation combined deterministic unit checks of tileToScreen and the plot hash with node --check syntax checks and the full pytest suite; manual browser verification is documented as a remaining step before flipping the default.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

Phase 1 — Scaffold

Created isocity.js with the same IIFE/public API as skyline.js, a 2:1 diamond projection, a tile grid sized to company count, and drawImage blitting of preloaded ground tiles.

Phase 2 — State Mapping

Assigned each company a stable plot via a deterministic hash of company_id, mapped emissions to smog density, compliance shortfall to a red distressed tint, and active abatement measures to clean-factory sprite swaps.

Phase 3 — Layering & Effects

Implemented back-to-front painter's draw order (sorted by col+row), reused the particle system for smog and action effects, and preserved the existing trigger API for abatement, offset, and year-transition events.

Phase 4 — Feature Flag & Parity

Introduced the ?renderer=isocity flag, swapped the init call in game.html, coop.html, and summary.html, and confirmed overlay labels still update correctly.

07 / Findings & Recommendation

Findings & Recommendation

The renderer is functionally complete and safely behind a flag. It meets the Sprint 3 objective: a deterministic isometric city that reflects engine state without regressing the shipped skyline.

Recommendation: confirm visual parity through manual testing across solo, multiplayer, and co-op sessions, then flip the default to isocity in Sprint 4 after the per-screen retro sweep is complete.

Implementation Path

09 / Risks / Open Questions

Risks / Open Questions

Risks

  • isocity has not yet been visually verified in a real browser session; manual testing is required before flipping the default.
  • Large multiplayer rooms may stress the painter's algorithm if the 16-plot cap proves too low or too high.

Open Questions

  • What is the optimal tile anchor offset for crisp sprite placement at different DPRs?
  • Should smog density animate smoothly or snap to discrete levels?
  • When should skyline.js be physically removed from the repository?

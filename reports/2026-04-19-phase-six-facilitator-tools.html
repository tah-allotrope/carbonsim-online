03 / Math / Algorithm Used

Math / Algorithm Used

Session pause and deadline extension

When the facilitator pauses, the engine records paused_from and the current phase in phase_before_pause. On resume, the pause duration is added to the phase deadline so participants lose no decision time. This prevents unfair time pressure from facilitator pauses.

Facilitator force-advance

Each call to force_advance_phase transitions one phase step: Year Start → Decision Window → Compliance → Next Year Start or Session Complete. The function logs a phase_force_advanced audit event so the facilitator's manual interventions are traceable in post-session analysis.

Participant status tracking

Every participant action (activate_abatement, buy_offsets, submit_auction_bid, propose_trade, respond_trade) increments decision_count_this_year for that company. The counter resets to zero when a new year starts, giving facilitators a live read on who is engaged and who may need prompting.

Export and summary

export_session_data serializes the full engine state: companies with all year results, auctions with bids and clearing outcomes, trades with statuses, ranked companies sorted by cumulative penalties, and the complete audit log. build_session_summary adds aggregated metrics (total penalties, trades completed, auctions cleared, average clearing price) and contextual facilitator notes that adapt to session phase.

06 / Limitations / Second-Best Alternative

Limitations / Second-Best Alternative

  • Participant status is activity-based, not connection-based. The participant_status tracker counts decisions and records last action time, but it does not detect whether a browser tab is still connected. A future V2 enhancement could use WebSocket heartbeat or oTree live ping to track actual connection state.
  • Export is JSON-only in the engine layer. The export_session_data function returns a plain Python dict. Download formatting (CSV, Excel) and the oTree built-in data export are separate concerns that the facilitator panel handles client-side for now.
  • No replay or undo. Facilitator actions (pause, resume, advance) are audit-logged but not reversible. If a facilitator accidentally advances a phase, the only recovery is manual reconfiguration or starting a new session. This is acceptable for V1 workshops but should be revisited for production use.
  • The session summary provides facilitator notes, not automated recommendations. The notes are contextual (e.g., "no abatement measures activated yet") but do not generate pedagogical advice automatically.
  • The second-best alternative would have been using oTree's built-in admin pages for session control. However, those pages are not customizable enough to show participant health, market summaries, or debrief notes. Building a dedicated facilitator panel gives full control over the workshop flow and presentation.
07 / Errors / Warnings / Flags

Errors / Warnings / Flags Encountered

  • Initial test failures for force_advance and decision counter reset. Two tests failed on the first run: test_force_advance_from_compliance_to_next_year (needed to step through decision window first via force advance) and test_decision_counter_resets_on_new_year (needed to use force_advance instead of advance_state with time compression). Both were fixed by restructuring the test sequences to use explicit phase steps rather than relying on timer-driven transitions.
  • LSP type errors for oTree imports are expected and harmless — oTree is installed in the project virtual environment but not in the system Python used by the language server.
  • No blocking errors. All 41 tests pass cleanly.
08 / Open Questions / Next Phase Seeds

Open Questions / Next Phase Seeds

  • Phase 7: Pilot scenario design and bot support. The next phase should calibrate the cap decline, abatement costs, and penalty levels for a compelling three-year arc, and add optional bot participants for liquidity support when human count is low.
  • Client-side export download. The facilitator panel currently shows export JSON in a pre-formatted block. A future iteration should offer one-click download as a .json file.
  • Connection health monitoring. Track actual WebSocket/connection state rather than just last-action time, so the facilitator can see which participants are truly disconnected versus just inactive.
  • Facilitator undo or checkpoint. A session snapshot and rollback capability would make the system more robust for workshop settings where mistakes happen.
  • Mobile usability for facilitators. The facilitator panel works on desktop but should be tested and possibly optimized for tablet use during live workshops.

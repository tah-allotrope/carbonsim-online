Executive Summary

Sprint 1 shifted CarbonSim's visual mood from a muted retro-parchment look to the bright, rounded, gradient Kairosoft register — the cheapest, highest-visibility step of the four-sprint Kairosoft visual roadmap. Because the stylesheet is fully tokenized, the change was almost entirely a token + component restyle in one CSS file, plus a documentation rewrite and a consistency sweep of inline colors across the four screens. It is frontend-only — no engine, API, or gameplay code was touched, so the 116-test suite is unaffected.

All three phases are complete, verified live in the browser, and pushed to master. Two items were intentionally deferred: a rounded-pixel display webfont (needs a licensed binary; Press Start 2P retained) and a full accessibility/contrast re-measurement (flagged for Sprint 4). Recommendation: proceed to Sprint 2 (building sprite art), which the new palette and a fresh sprite spec now unblock.

Background & Objective

CarbonSim is a single + multiplayer carbon-compliance tycoon with an isometric pixel-art city. Research (research/2026-06-14_kairosoft-visual-style.md) found the gap to a Kairosoft look is "art and juice," not the rendering engine — and that the quickest, highest-impact lever is palette + UI chrome. Sprint 1's objective: establish the bright Kairosoft mood and the target palette that every later art sprint must match.

Inputs & Scope

  • Plan: plans/2026-06-14-kairosoft-palette-ui-plan.md (Sprint 1 of the roadmap index).
  • Research: research/2026-06-14_kairosoft-visual-style.md; gap analysis reports/2026-06-14-kairosoft-visual-style-gap-analysis.md.
  • Touched: web/css/style.css, docs/design-language.md, and inline styles in web/index.html, game.html, coop.html, summary.html.
  • Out of scope: building sprites (Sprint 2), citizens (Sprint 3), floating feedback (Sprint 4), the canvas renderer, and any engine/gameplay logic.

Assumptions & Constraints

  • Token names kept stable so a value swap propagates everywhere with no markup edits.
  • No build step (vanilla CSS); fonts must be self-hosted (offline-friendly) — ruled out a CDN rounded font.
  • Accessibility floor: 4.5:1 body / 3:1 large-bold; color never the sole indicator.
  • Full reskin (single design system), not a theme toggle.

Methodology

Each phase followed: implement → verify live in Claude Preview (computed-style readback + screenshot) → generate a phase report → commit & push. Phases were scoped to be cleanly separable so each commit is meaningful and reversible.

Phase 02 — Rounded chrome, gradients & bubble

Beveled components became pill buttons (gradient + colored bottom edge), 16px rounded gradient cards/modals/event-cards, white inputs with a 3px focus ring, gradient badge pills, and a reusable .bubble speech-pill for Sprint 4.

Phase 03 — Design doc & cross-screen consistency

Rewrote docs/design-language.md for the Kairosoft system and added the Building Sprite Spec that Sprint 2 builds against. Swept the four screens, replacing old muted-teal/neon inline literals (player-row highlights, screen-flash, XP-badge border, the summary Chart.js palette) — grep confirms none remain.

Findings & Recommendation

  • Tokenization paid off: one CSS file drove an app-wide mood change with no markup edits.
  • Verified live: azure pill buttons, 16px soft-shadow cards, gradient badges; game screen renders with the city painting (18,525 non-blank px) and all sections intact.
  • Tradeoff accepted: bright-but-legible azure #1f93c7 over a louder hue, to keep white button-text contrast.
  • Recommendation: proceed to Sprint 2 — building sprite art; the palette and sprite spec are the prerequisites and are now in place.

Implementation Path

  1. Decide asset strategy for Sprint 2 (author original pixel art vs. CC0/CC-BY packs; never Kairosoft IP) — gates the art work.
  2. Sprint 2: produce buildings to the new Building Sprite Spec → register in web/assets/manifest.json → wire isocity.js drawPlotStructure by sector/state.
  3. Then Sprint 3 (animated citizens + frame-anim core) and Sprint 4 (floating feedback + perf/a11y, including the deferred contrast pass).
  4. Quick win anytime: drop in a rounded-pixel OFL display font (add .ttf + @font-face + ATTRIBUTIONS.md; point --font-display).
